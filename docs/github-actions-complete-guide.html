📋 Complete Guide: GitHub Actions, CI/CD & Interview Questions

Comprehensive explanation of GitHub Actions workflows, CI/CD concepts, and answers to common interview questions.

📌 What is CI/CD? (Simple Explanation)

Think of it like an automated quality checker for your code:

  • CI (Continuous Integration): Every time someone pushes code, automatic tests run to check if the code is correct before merging
  • CD (Continuous Deployment/Delivery): If tests pass, the code is automatically deployed to the live website

Real-world analogy: Before shipping a product, you don't just pack and send it. You test it multiple times. CI/CD automates this testing process.

In this project: We use CI with GitHub Actions to run Playwright tests and CodeQL security scans on every push and pull request.

🔍 What is GitHub Actions? (Simple Explanation)

In simple terms: GitHub Actions is like a robot that watches your code repository. When something happens (like you push code), the robot automatically runs tasks for you.

Key Components:

  • Workflow: Instructions written in YAML file (stored in .github/workflows/) that tell the robot what to do
  • Event/Trigger: What makes the robot wake up (push code, create pull request, scheduled time)
  • Job: A set of related tasks the robot needs to do
  • Step: Individual commands within a job (like "run npm install")
  • Action: Pre-made tools that other people shared (like downloading code, installing Node.js)
  • Runner: The actual computer (Linux, Windows, macOS) where the robot runs

1️⃣ Playwright Tests Job - Detailed Explanation

File: .github/workflows/playwright.yml

Purpose: Test your website in Chrome, Firefox, and Safari automatically whenever code changes.

📊 What Happens Step-by-Step

Developer pushes code → GitHub triggers workflow → Get code → Install Node.js → Install packages → Install browsers → Run tests in 3 browsers → Report results

⚙️ Detailed Step Explanation:

Step 1: Checkout Repository

Command: actions/checkout@v4

Simple explanation: Download your code from GitHub to the robot's computer so it can work with it.

Why: The robot needs the code to test it.

Step 2: Install Node.js

Command: actions/setup-node@v4 with lts/* (latest stable)

Simple explanation: Install Node.js (JavaScript engine) which Playwright needs.

Why: Playwright is written in JavaScript and needs Node.js to run.

Step 3: Install Project Dependencies

Command: npm ci (Clean Install)

Simple explanation: Download all libraries your project needs (from package.json)

Why npm ci vs npm install?

  • npm install - Used locally, may update packages
  • npm ci - Used in CI/CD, ensures exact same versions always

Steps 4-6: Install Browsers

Commands:

  • npx playwright install chromium --with-deps (15 min timeout)
  • npx playwright install firefox --with-deps (25 min timeout)
  • npx playwright install webkit --with-deps (25 min timeout)

Simple explanation: Download real browsers so Playwright can test your website in them.

Why 3 browsers? To ensure your website works for everyone (Chrome users, Firefox users, Safari/Apple users)

What is --with-deps? Also download system libraries these browsers need on Linux

Step 7: Run Tests

Command: npx playwright test --retries=2

Simple explanation: Actually run your test files and see if they pass.

What is --retries=2? If a test fails, try it 2 more times. This handles "flaky" tests that randomly fail due to timing issues.

Tests run in: All 3 browsers simultaneously (parallel execution for speed)

Step 8: Save Test Report

Command: actions/upload-artifact@v4

Simple explanation: Save the test report to GitHub so you can see details about which tests passed/failed.

Special feature: Runs even if tests fail (important for debugging)

Report kept for: 30 days

⏱️ Total Time: ~10-12 minutes for entire Playwright workflow

2️⃣ CodeQL Security Analysis - Detailed Explanation

File: .github/workflows/codeql.yml

Purpose: Scan your code for security vulnerabilities without running it.

📊 What Happens

Developer pushes code → GitHub triggers → Get code → Install Node → Setup CodeQL → Analyze code → Report security issues

⚙️ Detailed Explanation:

Step 1: Get Code

Command: actions/checkout@v4

What it does: Download code to analyze

Step 2: Install Node.js

Command: actions/setup-node@v4

What it does: Install Node so CodeQL understands JavaScript/TypeScript

Step 3: Initialize CodeQL

Command: github/codeql-action/init@v3 with languages: javascript

What it does: Set up CodeQL to analyze JavaScript code

What is CodeQL? A code analysis tool that finds security problems by looking for dangerous code patterns

Step 4: Install Dependencies

Command: npm ci

What it does: Install libraries so CodeQL can see how your code uses them

Step 5: Analyze Code

Command: github/codeql-action/analyze@v3

What it does: Find security vulnerabilities

What it finds: SQL injection, XSS attacks, password leaks, unsafe code patterns

⏱️ Total Time: ~8-15 minutes (longer because deep analysis is complex)

🎯 Interview Questions & Answers: CI/CD & GitHub Actions

🔹 Basic Concepts

Q1: What is CI/CD and why is it important?

A: CI/CD is about automation:

  • CI (Continuous Integration): Automatically test code after each commit
  • CD (Deployment/Delivery): Automatically deploy tested code

Why important:

  • Find bugs immediately (before users see them)
  • Reduce manual testing time by 90%
  • Deploy to production faster
  • Consistent, quality releases
  • Team can focus on feature development instead of testing
Q2: What are the benefits of using GitHub Actions vs other CI/CD tools (Jenkins, Travis CI, CircleCI)?

A: GitHub Actions advantages:

  • Built into GitHub: No external account needed
  • Free for public repos: Unlimited minutes
  • Easy setup: Just create YAML files in your repo
  • Marketplace: Thousands of pre-built actions to reuse
  • No separate interface: Everything in GitHub repo
Q3: What are the main components of GitHub Actions workflow?

A:

Component What It Is Example
Workflow YAML file with automation rules playwright.yml
Trigger/Event What starts the workflow push, pull_request
Job Container with multiple steps test, analyze
Step Individual command or action npm install
Action Pre-built reusable code actions/checkout@v4
Runner The computer running the job ubuntu-latest

🔹 Workflow & Triggers

Q4: Difference between "push" and "pull_request" triggers?

A:

  • Push: Runs when code is pushed to main/master branch (after PR is merged)
  • Pull Request: Runs when PR is created/updated (before merging)

In practice: Push on main ensures no broken code reaches production. PR checks ensure code is ready before merging.

Q5: What's the difference between job-level and step-level timeout?

A:

  • Job timeout: Maximum time entire job can run (60 minutes in playwright.yml)
  • Step timeout: Maximum time single step can run (15 min for Chromium install, 25 min for Firefox)

If exceeded: Job/step automatically stops and marked as failed

Q6: What does "if: ${{ always() }}" do?

A: Makes a step run even if previous steps fail.

Example: Upload test report runs even if tests fail, so you can debug what went wrong

Other conditions:

  • if: success() - Run only if all previous steps passed
  • if: failure() - Run only if any step failed
  • if: always() - Run regardless of pass/fail
Q7: Can you run multiple jobs in parallel?

A: Yes, by default all jobs run in parallel unless you add needs keyword.

In this project: Playwright Tests and CodeQL run at same time (parallel)

Total time is: Longest job (~15 min), not sum of both

🔹 Testing & Dependencies

Q8: npm ci vs npm install - when to use each?

A:

Command Purpose When to Use
npm install Install and may update package versions Local development
npm ci Install exact versions (clean install) CI/CD pipelines

Why? In CI/CD, we want consistency. Everyone should test same versions.

Q9: What is Playwright and why use it in CI/CD?

A: Playwright is a tool that automates browser testing across Chrome, Firefox, Safari.

Why use in CI/CD:

  • Test real user interactions (clicking, typing, navigation)
  • Works on multiple browsers
  • Catches UI bugs that unit tests miss
  • Ensures actual users will have good experience
  • Fast and reliable
Q10: What does "--retries=2" do?

A: Makes failing tests retry up to 2 additional times.

Why? Some tests fail randomly due to timing (network delay, element not ready). Retrying catches temporary failures.

Example: Test fails first time (network slow), passes on retry (network recovered)

Q11: What is CodeQL and what security issues does it detect?

A: CodeQL is GitHub's static analysis tool. It reads code without running it to find vulnerabilities.

Security issues detected:

  • SQL Injection (hackers add bad SQL commands)
  • Cross-Site Scripting (XSS attacks)
  • Sensitive data exposure (passwords in code)
  • Insecure encryption
  • Unsafe command execution
  • Weak authentication

🔹 Advanced Topics

Q12: What are GitHub Actions Secrets and how do you use them?

A: Secrets are encrypted environment variables for sensitive data.

Why? Never put passwords, API keys, tokens in code (they're public on GitHub)

How to use:

steps: - run: echo "${{ secrets.MY_PASSWORD }}"

Setup: GitHub repo Settings → Secrets and variables → New secret

Q13: How do you cache dependencies to speed up workflows?

A: Use actions/cache@v3 to save npm packages between runs.

Benefits:

  • First run slower (downloads packages), subsequent runs faster (uses cache)
  • Reduces GitHub server load
  • Faster deployments
- uses: actions/cache@v3 with: path: ~/.npm key: ${{ runner.os }}-${{ hashFiles('**/package-lock.json') }}
Q14: What is a GitHub Actions matrix and how is it used?

A: Matrix runs a job multiple times with different configurations.

Example: Test on Node 16, 18, 20

strategy: matrix: node-version: [16, 18, 20] steps: - uses: actions/setup-node@v4 with: node-version: ${{ matrix.node-version }}

Result: Job runs 3 times, once per version

Q15: How do you trigger a workflow manually?

A: Add workflow_dispatch trigger to YAML.

on: push: workflow_dispatch:

Then: Go to GitHub Actions tab → Select workflow → "Run workflow" button

Q16: How do you organize multiple workflows in a project?

A: Best practices:

  • Create separate files for different purposes (test.yml, deploy.yml, security.yml)
  • Use descriptive names
  • Use job dependencies with needs keyword
  • Reuse common logic with composite actions or workflow files
  • Document triggers and purpose in comments

In this project: We have separate files for testing and security (good practice)

Q17: How do you debug a failing workflow?

A: Debugging steps:

  • Click failed run in Actions tab
  • Find the step with red ❌ mark
  • Read error message carefully
  • Check if dependencies installed
  • Verify environment variables/secrets set
  • Test commands locally first
  • Enable debug logging: set ACTIONS_STEP_DEBUG=true secret
Q18: What's the difference between GitHub Actions artifacts and releases?

A:

  • Artifacts: Temporary (30 days) workflow outputs, stored in Actions tab
  • Releases: Permanent version releases for users to download

In this project: Test reports are artifacts (for debugging)

Q20: How do you handle workflow failures and notifications?

A: GitHub automatically:

  • Shows ❌ red status if workflow fails
  • Sends email to committer
  • Blocks PR merge if required checks fail

Additional options:

  • Slack integration for notifications
  • Custom notifications with actions
  • Set branch protection rules
Q21: What are branch protection rules and how do they work with workflows?

A: Branch protection rules prevent code merge until requirements are met.

Common requirements:

  • All workflow checks must pass (Playwright Tests ✅ and CodeQL ✅)
  • Code review approval required
  • No commits directly to main (must use PR)

Setup: GitHub repo Settings → Branches → Add rule

Q22: What's the difference between `run` and `uses` in workflow steps?

A:

  • run: Execute a shell command (your own code)
  • uses: Use a pre-built action from marketplace
# Using run - run: npm install # Using uses - uses: actions/checkout@v4
